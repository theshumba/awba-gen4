2 · The awba.app cutover

Do this only after the site is live at theshumba.github.io/awba-gen4/ and you're ready to switch the custom domain. Adding the CNAME file before DNS resolves triggers GitHub's "domain not properly configured" error.

Step 1 — at the awba.app DNS host, add four A records for the apex (@)

@   A   185.199.108.153
@   A   185.199.109.153
@   A   185.199.110.153
@   A   185.199.111.153

Step 2 — add the www subdomain as a CNAME

www CNAME theshumba.github.io

Step 3 — tell GitHub Pages the domain (one command, from the repo root)

printf 'awba.app\n' > CNAME && git add CNAME && git commit -m "Point Pages at awba.app" && git push

Then in the repo: Settings → Pages → Custom domain = awba.app, and tick Enforce HTTPS once the certificate provisions — it can take a few minutes to an hour.

Step 4 — propagation check

dig +short awba.app            # → the four 185.199.x.153 addresses above
dig +short www.awba.app        # → theshumba.github.io (then the same four IPs)
curl -sI https://awba.app/     # → HTTP/2 200, and a valid TLS cert (no warning)

Every path in the app is relative, so the exact same commit serves correctly at the Pages subpath and at the awba.app root — no code change accompanies the cutover. The manifest's start_url/scope stay relative too.

3 · Licensing & the scholar gate

Clear Quran translation licensing — unresolved (yours; not a build blocker, never silently cleared)

Awba shows the Clear Quran (Dr. Mustafa Khattab) translation of the meaning verbatim, with the fixed source line and an unverified · pending review pill on every citation. The launch text was drawn against the quran.com / quranapi.pages.dev API, whose public translation endpoints have signalled withdrawal/deprecation for third-party redistribution — so relying on that API as a live source is not safe for a commercial launch, and redistributing the Clear Quran text itself likely needs a publisher licence from the rights holder. Before commercial launch: confirm a licence (or an explicitly permitted source) for the Clear Quran text, and stop depending on the withdrawn API. The app is already self-contained (the text is embedded verbatim, not fetched), so this is a rights question, not a technical one.

Scholar sign-off — the pack and what it unlocks

docs/scholar-pack.html is the printable, verbatim, fully-sourced review document — every verse, every hadith and the engine du'a, each marked pending review. Hand the scholar that one file. When it is signed off:

Removing the unverified · pending review pills is a future code change, listed here as such — it is not part of this release. Today the pill is a blanket, global posture (every citation, unconditionally). A signed pack would motivate either a global "reviewed" flip or a finer per-citation verified state; either way it is an engine/CSS change gated behind the actual sign-off. Until then, no surface may imply scripture has been cleared.
