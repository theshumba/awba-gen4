# Domain Pitfalls

**Domain:** Gamified Duolingo-style Islamic micro-learning PWA — zero-build vanilla JS, verbatim scripture, mercy-framed gamification (Awba Gen-4 rebuild of Josh's Gen-3 MVP)
**Researched:** 2026-07-11
**Sources examined directly:** `/Users/theshumba/Downloads/AWBA APP/_MVP-BUILD/learn.html`, `shared/awba-engine.js` (465 lines), `shared/awba-engine.css` (326 lines), `lessons/u1-m1.html`, `reviews/u1-review.html`, `00_BUILD-RECORD.md` §7 changelog (v1.1–v1.5)

## Critical Pitfalls

Mistakes that cause rewrites or major issues.

### 1. Treating the "phone card" as the layout, not a preview of one

**What goes wrong:** Josh's MVP renders everything inside a fixed `.phone{width:380px;max-width:96vw;height:788px;max-height:94vh}` card, centered on a colored backdrop (`body{display:flex;align-items:center;justify-content:center}`), with every screen inside it absolutely positioned (`.hero{position:absolute;inset:0}`, `.stage{position:absolute;inset:56px 0 96px}`, `.foot{position:absolute;left:0;right:0;bottom:0}`). This is a **prototype-in-a-frame**, not a true responsive mobile layout — it was built for Josh to review on a laptop, not for a phone browser to actually run full-bleed. PROJECT.md's active requirement is "Fully responsive (phone-first, works desktop) + installable PWA shell." If Gen-4 ports the CSS near-verbatim, a real phone browser (or the installed PWA) will show a small rounded rectangle floating on a colored background with wasted margin on all sides — the opposite of "world-class" and the opposite of what an installed PWA should look like (which needs true full-viewport, safe-area-aware layout).
**Why it happens:** The fixed-frame approach was the fastest way to get a working one-shot MVP with predictable dimensions, and it visually "reads" as an app in a screenshot. It's an easy trap to inherit unmodified during a "just make it prettier" rebuild.
**Consequences:** Real device testing looks broken/unpolished; PWA installed icon opens to a shrunk floating card; iOS/Android status bar and home-indicator safe areas collide with the fixed 788px frame; desktop "works" only by accident (giant colored void around a tiny card) unless deliberately designed as a desktop state.
**Prevention:** Rebuild the shell as true full-viewport (`100%`/`100dvh` root, safe-area-inset padding, no fixed pixel container) for mobile, and design an explicit, intentional desktop treatment (e.g., centered card *as a deliberate desktop-only presentation*, not the mobile layout literally scaled). Keep the absolute-positioned screen-swap pattern (hero/stage/foot) — it's a reasonable SPA-in-a-page technique — but re-derive it against real viewport units, not a fixed box.
**Detection:** Open the rebuilt shell on an actual phone (or responsive-mode at 100% zoom, no forced frame) before any polish pass. If you see visible page-background gutters around a rounded card, this pitfall is live.
**Phase:** Should be resolved at the very start of the shell/engine phase — every other phase (lesson porting, reward choreography, PWA) builds on top of this container decision, so fixing it late means re-touching all 19 pages' inline layout assumptions.

### 2. Breaking the `AwbaLesson(cfg)` / `AwbaReview(cfg)` global-function contract

**What goes wrong:** All 15 lesson files and 4 review files are static HTML pages that do `<script src="../shared/awba-engine.js"></script>` then `<script>AwbaLesson({...})</script>` (or `AwbaReview({...})`) as **plain global function calls**, evaluated synchronously in document order. `AwbaLesson`/`AwbaReview` are bare function declarations on `window`, not exports from a module. Content authors (Josh) reference `AW.cite(...)` *inside the cfg object literal itself* (e.g. `revelation:'...'+AW.cite('hujurat-49-15','al-Ḥujurāt 49:15')`), which means `AW` must already exist and script execution order must be preserved exactly: engine.js loads and defines `AW`/`AwbaLesson`/`AwbaReview` on `window` → the inline lesson script's object literal evaluates (calling `AW.cite` eagerly at parse time) → `AwbaLesson(cfg)` runs. Any "modernization" that converts the engine to an ES module (`<script type="module">`, `export function AwbaLesson`), adds `defer`/`async` to the engine `<script>` tag, or bundles/minifies in a way that changes load timing, will break every one of Josh's 19 pages simultaneously and silently (blank page, `AwbaLesson is not defined` or `AW.cite is not a function` in console — nothing visible to a non-technical reviewer opening the file).
**Why it happens:** It's the natural instinct of a "high polish rebuild" to clean up global-scope JS into modules, which is good practice in general but is exactly the kind of "we didn't change the content, we just modernized the plumbing" change that silently invalidates the content contract.
**Consequences:** All 15 lesson files + 4 reviews stop rendering; Josh's zero-build "open the file directly to review" workflow (a named project constraint) breaks entirely.
**Prevention:** Treat `window.AwbaLesson(cfg)` / `window.AwbaReview(cfg)` as a frozen public API. Internally refactor the engine however you like (split into modules bundled at author-time if truly needed, cleaner internal structure, TypeScript-authored-then-compiled, etc.) but the *shipped* `awba-engine.js` that lesson/review pages `<script src>` must still expose plain global synchronous functions with the exact same call signature, and must still make `AW.cite`/`AW.*` available synchronously before the inline cfg-literal script runs. Do not add `defer`/`async`/`type="module"` to the engine `<script>` tag referenced by lesson/review pages unless every single data file is re-verified to still load `AW` in time.
**Detection:** After any engine change, run `node --check` on the engine file (syntax only — insufficient alone, see Pitfall 6) **and** headless-load every one of the 19 pages, asserting `document.title` is set and the root `#phone`/`#root` node is non-empty. Josh's own verification log already does exactly this ("`node --check` on engine + all 18 pages' inline scripts: PASS · headless load of all 18 pages: zero page errors") — keep this as a standing regression gate, not a one-time check.
**Phase:** Engine/shell-rebuild phase — must be a hard contract test before any lesson-porting phase begins, and re-run after every engine change thereafter.

### 3. Assuming localStorage alone can honor "un-loseable noor, unbreakable returns streak"

**What goes wrong:** The entire mercy-mechanics promise ("un-loseable noor," "unbreakable returns streak," "NEVER BREAKS" pill copy) is backed by nothing but `localStorage` (`AW.S.get/set` with an `awba_` prefix). iOS Safari applies a 7-day cap on script-writable storage (localStorage/IndexedDB) for sites used as regular Safari tabs (ITP's anti-tracking storage eviction, shipped iOS 13.4+) — if a learner uses Awba as a bookmarked Safari tab rather than installing it to the home screen, and doesn't open it for 7+ days, iOS can wipe `noor`/`returns`/`stars`/`days` out from under them, directly breaking the product's core promise. Home-screen–installed PWAs (standalone display mode) get their **own** storage counter separate from Safari's, and Apple has stated it does not expect first-party site data to be deleted for these — but this protection only applies once a user has actually installed the PWA, which most users won't do on first visit.
**Why it happens:** Vanilla static-site + localStorage-only is the simplest possible persistence model, and it was explicitly chosen for v1 ("localStorage-only progress, no accounts... ships fastest"). The risk is invisible in normal development/testing (desktop browser sessions don't hit the 7-day idle threshold) and only shows up for real users weeks later.
**Consequences:** A learner who returns after a week+ away — the exact "amber, never red / it's good to see you again" scenario the product is designed to be gentle about — could find their progress silently gone, which is the single worst possible failure mode for a product whose entire pitch is "nothing is ever lost."
**Prevention:** (a) Actively prompt/encourage "Add to Home Screen" early (the PWA install requirement already planned should be framed partly as a *data-safety* feature, not just a convenience); (b) treat this as a known v1 limitation and say so explicitly somewhere in-product (e.g., in the returns sheet copy) rather than over-promising "never breaks" without caveat; (c) keep an export/import-progress affordance in mind as a cheap safety net (a "backup my light" button that dumps the `awba_*` keys to a shareable string/file) even if the account/backend system stays out of scope for v1; (d) do not silently let this slide into v2 as "someone else's problem" — log it as a known limitation for Josh/Melusi the same way the Quran-API licensing flag was logged.
**Detection:** Can't be caught by normal QA (requires real 7+ day idle testing). Treat as a known-risk item to flag explicitly in documentation rather than something to "test away."
**Phase:** Storage/state-model phase (early) for the design decision; PWA/install phase for the mitigation (install prompting); content/copy phase for setting honest expectations in the streak/noor sheet copy.

### 4. Re-deriving lesson content from source atoms instead of porting Josh's verbatim HTML/JS

**What goes wrong:** The course has explicit, deliberate content holds baked into Josh's lesson files, not into the underlying atom documents: **U4-03 is absent entirely** (a full hold), U3-13 (cow-veneration anecdote) is deliberately not surfaced, U3-16 is taught only as "one true religion" with the naming-frame held back, and several group-namings in U1/U2 are held. If Gen-4 is built by an AI/dev regenerating lesson copy from `AWBA_Research_and_Pedagogy_Master.md` or the atom-level source documents (which likely contain the full, unredacted material) rather than literally porting Josh's already-redacted HTML strings, there is a real risk of the holds being silently reintroduced — because the redaction lives in what Josh chose to *omit* when writing the lesson, and that omission is invisible if you're working from the atom source rather than from his output.
**Why it happens:** "Rebuild at higher polish" invites regenerating text for better prose/flow, which is reasonable for *presentation* but dangerous for *content selection* — a well-meaning rewrite of a paragraph can accidentally pull in a detail from the source atom that Josh deliberately left out.
**Consequences:** Scholar-sensitive or held-back religious content ships without scholar review, undermining the entire "scholar-gated, verbatim, sourced" content-integrity constraint.
**Prevention:** Port Josh's lesson `beats[]` arrays, `refs{}`, and `terms{}` objects verbatim (structurally and textually) as the content baseline. If any copy is rewritten for polish, diff it explicitly against Josh's original per-lesson HTML strings and re-check it against the hold list in `00_BUILD-RECORD.md` §4 ("Sensitive atoms") before shipping — never regenerate a lesson "from atoms" as if starting fresh. Preserve the `unverified · pending review` pills on every citation exactly as implemented (`AW.sheetRef` always renders this pill; do not let a "cleaner" citation sheet redesign quietly drop it).
**Detection:** Content-review checklist per lesson: (1) diff against source lesson file for the specific held atoms/namings listed in §4/§5 of the build record, (2) confirm every `refs{}` entry still carries `pending review` in its `src` field, (3) confirm U4-03 remains entirely absent from the U4 lesson set.
**Phase:** Content-porting phase — should be a named verification step per lesson/unit, not a one-time global check, since it's easy to introduce per-lesson during iterative polish.

### 5. Letting reward choreography or confetti compete with scripture

**What goes wrong:** The engine's `resolve()` function fires `PERFECT` confetti (30 particles, full-screen `rgba(37,54,206,.94)` overlay) on any 3-answer combo, and the verdict screen fires confetti unconditionally (26 or 14 particles) after *every* lesson. Because quiz beats (`mc`/`tf`/`tile`) can sit structurally close to `verse` beats in a lesson's `beats[]` array (the content author controls ordering, and several lessons place a headline verse card immediately before or after a knowledge check), a rebuild that changes pacing/animation timing, or that adds page-transition motion "for polish," risks a scenario where confetti or a full-screen celebration overlay visually overlaps or immediately follows a scripture card still fading out — a direct violation of the locked pedagogy rule ("confetti never over ayahs"). The existing implementation is careful about this only by virtue of `beats` sequencing keeping quiz-resolution feedback in the *footer* (not full-screen) except for the rare 3-combo `PERFECT` overlay and the always-full-screen verdict screen (which happens *after* the beats loop ends, never mid-scripture) — a rebuild that adds new celebratory moments (e.g., confetti on a `verse` beat's "Continue," or richer per-beat transition effects) could easily reintroduce this exact violation the source pedagogy brief warns against.
**Why it happens:** "World-class motion + polish... Duolingo-grade" as a stated goal pulls naturally toward adding *more* delight moments, and it's easy to add a celebratory flourish to a screen transition without checking what content type it's leaving or entering.
**Consequences:** Breaks the product's explicit "mercy framing" pedagogy constraint (no casino juice over scripture) — this is a named, locked design principle, not a nice-to-have.
<br>
**Prevention:** Any new animation/celebration effect must be checked against a simple rule before shipping: *never let a confetti/full-screen-celebration trigger fire while a `verse` beat (or its immediate transition) is on screen or fading out.* Keep celebration moments anchored to answer-resolution and end-of-lesson screens only (as today), not to scripture-display moments. When adding page-transition polish, verify the transition into/out of `verse` beats specifically uses calmer motion than quiz-answer transitions.
**Detection:** Manual walkthrough of every lesson that places a `verse` beat adjacent (before/after) to a quiz beat, watching specifically for confetti/overlay bleed timing.
**Phase:** Reward-choreography/motion-polish phase — flag explicitly as a design constraint before motion work begins, and spot-check during that phase's QA.

### 6. No schema validation on the lesson/review `cfg` contract — silent failures when porting 15 files

**What goes wrong:** `AwbaLesson(cfg)` and `AwbaReview(cfg)` have no runtime validation of `cfg` shape. The beat renderer is a chain of `if(it.t==='read'){...return;} if(it.t==='frame'){...return;}` etc. — an unrecognized `t` value (a typo like `'reed'`, or a beat type introduced in one file but not the shared switch) falls through **every** branch and renders nothing for that step, with no error thrown and no console warning. Similarly, required per-type fields (`it.o`/`it.c` for `mc`, `it.solution` for `tile`, `refs[id]` existing for every `AW.cite(id,...)` call, `terms[id]` existing for every `data-term="id"` reference) are accessed directly with no existence checks — a missing `refs` entry means `AW.sheetRef` does nothing when tapped (dead citation chip, not a crash), and this is invisible unless someone manually taps every citation chip in every lesson.
**Why it happens:** The engine was built one-shot for speed; `node --check` (Josh's existing verification step) only catches JavaScript *syntax* errors, not *semantic* mistakes in the config data (wrong beat type, orphaned citation id, missing required field for a beat type). Porting 15 lesson files by hand (or regenerating them) is exactly the situation where a copy-paste error, an off-by-one in a `refs` key, or a typo'd beat type is likely and easy to miss in a visual skim.
**Consequences:** A lesson ships with a silently blank step, a dead citation chip, or a non-functional tile-builder — degrading trust in a product whose whole pitch is polish and care, and it may not surface until a real learner hits that exact lesson/step.
**Prevention:** Build a small standalone validation script (Node, no framework needed — fits the zero-build constraint since it's a dev-time tool, not shipped) that loads each lesson/review data object (extract the `cfg` literal, or refactor data files to export a plain object the validator can `require`/`import`) and asserts: every `beats[].t` is one of the known types; every type-specific required field is present; every `data-term`/`AW.cite` id referenced in `html`/lens strings has a matching entry in `terms{}`/`refs{}`; every `refs{}` entry has non-empty `ar`/`mean`/`src`. Run this for all 19 pages as a standing pre-ship gate, not just `node --check`.
**Detection:** Add the validator to the same verification pass Josh already does (`node --check` + headless load) — extend it rather than replacing it. Manually tap every citation chip and term in a lesson during QA as a fallback until the validator exists.
**Phase:** Content-porting phase — build the validator before or alongside porting the first lesson, so it catches errors as they're introduced rather than at the end of a 15-lesson port.

## Moderate Pitfalls

### Regressing Josh's v1.1–v1.5 owner-review fixes during a full visual rebuild

Josh's build record §7 shows five rounds of real, specific bugs fixed live — a "high polish rebuild" that touches layout/CSS broadly is exactly the kind of change likely to reintroduce them if they aren't carried forward as explicit regression checks:
- **v1.4 popup anchoring:** `placePop()` positions the node popup over the *tapped node's own position* (not screen-center), clamping to the row width and offsetting the popup's arrow (`--ax` CSS custom property) so it still points at the node when clamped. A rebuild that changes the node/path DOM structure (e.g., a different grid system, a portal-rendered modal, absolute-positioning relative to a different ancestor) must re-derive this math (`el.offsetLeft`, `row.clientWidth`) against the new structure — don't let it regress to "popup always centers regardless of which node was tapped," which was the exact v1.3 bug.
- **v1.2 hero/footer overlap:** `.hero` reserves 128px above the fixed-position footer button and centers content via the `before`/`after` `margin:auto` flex trick (with scroll fallback) specifically so long content (stat cards, big noor numbers, longer recap text) can never slide under the button. Any hero/footer redesign must re-verify this with the *longest* real content case (longest reflect model-answer, longest recap list), not just the shortest.
- **v1.3 review timer mercy mechanic:** timing out a review question **auto-skips and parks it** (not "let them answer late with reduced reward" — that was explicitly rejected as "a gimmick"); parked questions come back **untimed and for no noor** in a circle-back phase; a correct circle-back answer still lights its lamp and counts toward "X of Y named" (mastery ledger) but **caps the run at 2 stars** and contributes no noor (reward ledger). These are two decoupled ledgers (mastery vs. reward) — a simplified rebuild of the review runner could easily collapse them back into one, silently regressing the mercy framing.
- **v1.1 accordion + chest idempotency:** the 3-Lens depth beat opens *each lens individually* (not all three at once — that was the pre-fix bug); gift chests grant `+25` noor exactly once (`chest_<id>` boolean gate) and show "already opened" on re-tap. A rebuilt accordion/chest UI must preserve both behaviors, not just the visual style.

**Prevention:** Before starting visual/layout rebuild work, extract §7 of `00_BUILD-RECORD.md` into a literal regression checklist and re-verify each item against the new build (screenshot or short screen-recording per item), the same way Josh verified them originally.
**Phase:** QA/polish phase, but the checklist should exist from day one of the shell rebuild so it's checked continuously, not only at the end.

### RTL handled via CSS `direction` only, not the `dir` attribute or bidi isolation

Arabic text blocks (`.arabic`, `.dar`, `.r-ar`, `.g-ar`) are styled with `direction:rtl;text-align:right` in CSS, applied to `<div>`s sitting inside an otherwise `<html lang="en">` LTR document — there is no `dir="rtl"` HTML attribute anywhere in the reviewed files, and no `unicode-bidi: isolate`/`plaintext`. CSS `direction` affects box layout and text alignment but is a weaker signal to the Unicode Bidi Algorithm than the `dir` attribute; when Arabic spans sit next to Latin numerals (sūrah:ayah references like "al-Ḥujurāt 49:15" right above/near the Arabic block) or punctuation, subtle reordering bugs at segment boundaries are possible, especially as more languages/scripts get mixed in future content (Fiqh/Seerah/Qur'an courses are already planned as "coming soon").
**Prevention:** For the rebuild, set `dir="rtl"` (or `dir="auto"`) directly on Arabic-content containers in addition to the CSS direction/alignment rules, and consider `unicode-bidi: isolate` (or `plaintext`) on those containers so they don't inherit bidi context from surrounding LTR content. Verify with a citation/reference string that mixes Arabic, a Latin transliteration, and a numeral reference (exactly the pattern already used throughout, e.g. `al-Ḥujurāt 49:15`) renders in the correct visual order on both LTR and RTL host contexts.
**Phase:** Design-system/typography phase, before content porting begins (touches every scripture/citation component).

### Font coverage gaps for transliteration diacritics and Khattab-style bracket glyphs

Content uses (a) extensive scholarly Latin transliteration with diacritics — ʿ ʾ ā ī ū ḥ ṣ ṭ ẓ ḍ ġ (e.g. `ʿaqīdah`, `Ṣaḥīḥ`, `Ḥujurāt`, `an-Naḥl`) — which must render correctly in the body/display fonts (Inter/Poppins), and (b) the Clear Quran/Khattab translation convention of wrapping interpolated words in `˹...˺` (U+02F9 MODIFIER LETTER BEGIN HIGH TONE / U+02FA MODIFIER LETTER END HIGH TONE — IPA spacing-modifier glyphs repurposed as brackets), used throughout every verse translation (confirmed in `learn.html`'s `DAILY` array and lesson `refs{}`). Neither of these is guaranteed to be covered by every weight/subset of Inter or Poppins as loaded from Google Fonts; a missing glyph falls back to the browser's default font for that character, producing a visible font-mismatch or a tofu box mid-sentence in body copy that's otherwise meant to look uniformly polished.
**Prevention:** Before finalizing typography, render a "diacritic + bracket stress test" string (one that includes the full character set above) in the actual chosen font/weights and visually confirm no fallback/tofu. If gaps exist, either pick a font that covers Latin Extended Additional + Spacing Modifier Letters blocks, or self-host/subset a font known to include them (verify rather than assume — this is a MEDIUM-confidence flag based on general Google Fonts subsetting behavior, not a confirmed test of these specific fonts' current glyph tables).
**Phase:** Typography/design-system phase, before content porting.

### Google Fonts CDN link repeated per page + no explicit FOUC strategy for Arabic

Every one of the 19 pages independently declares the same `<link href="https://fonts.googleapis.com/css2?family=Poppins:...&family=Inter:...&family=Amiri:...&display=swap">`. Browser HTTP caching mitigates repeat network cost after the first page load, but each *fresh* page navigation (this is a genuine multi-page site, not an SPA — every lesson/review is a full page load) re-parses the font CSS and re-triggers `font-display: swap` behavior: text renders immediately in a fallback font, then swaps once Amiri/Poppins/Inter load. For Arabic-heavy screens (verse cards, citation sheets) this means the Arabic block may briefly render in a fallback font that likely lacks correct Arabic glyphs entirely (showing tofu/boxes before Amiri loads), not just a font-style flash — worse than a typical Latin FOUC.
**Prevention:** Preload the Amiri font file specifically on screens/pages where Arabic renders immediately on load (verse-heavy lesson opens, daily ayah on Learn page) using `<link rel="preload" as="font">`, or set `font-display: optional`/`block` specifically for Amiri (accepting a brief invisible-text window instead of a tofu-then-swap window) rather than `swap` for that one family. Since this is a zero-build static site, this is a cheap `<link>`-tag-level fix, not an infra change.
**Phase:** Shell/typography phase — apply the same fix to every page template at once (or centralize the `<head>` font-loading block if page templates get consolidated during the rebuild).

### Reward choreography is a fixed 4-screen sequence with no fast path

Every lesson completion — and there are 19 completable units (15 lessons + 4 reviews) — walks the same sequence: `verdict` (stars + stat cards + confetti) → tap "Claim your noor" → `rewardNoor` (full-screen noor count-up) → tap "Lovely" → `rewardReturns` (full-screen streak hero + weekly calendar) → tap "Keep it gentle" → `done` (recap + next-lesson handoff). This is 3 extra full-screen taps *after* the actual learning content ends, every single time, with no way to skip or compress it for a learner doing a quick review-replay (tapping a "REVIEW · improve your stars" node they've already completed) or a fast, confident learner moving quickly through the course. Over 15 lessons + 4 reviews this ceremony is experienced 19 times identically.
**Prevention:** Consider (for the roadmap/design phase, not necessarily v1 must-fix) a lighter-weight completion path for replays/reviews specifically (e.g., collapse noor+returns into a single combined screen, or make the sequence skippable via a persistent "skip ahead" affordance) without removing the ceremony for a first-time completion, where it's earned and appropriate. At minimum, flag this explicitly as a known choreography-fatigue risk for owner review rather than treating the current 4-screen sequence as untouchable "because Josh shipped it."
**Phase:** Reward-choreography/motion-polish phase — an explicit design decision point, not an oversight to silently fix or silently leave.

### iOS Safari overscroll bounce, tap-highlight, and standalone-mode edge cases

The current CSS sets `-webkit-tap-highlight-color:transparent` globally (good) but has no `overscroll-behavior` handling — on a real full-viewport rebuild (see Pitfall 1), the elastic bounce-scroll effect at the top/bottom of scrollable regions (`.stage`, `.lbody`, `.sheet`) will be visible on iOS, including inside an installed home-screen PWA (standalone mode does not suppress this automatically, and Safari's `overscroll-behavior` support for containing it is inconsistent). Additionally, none of the reviewed files include a `viewport-fit=cover` meta tag or safe-area-inset handling, which will matter once the fixed-phone-frame is replaced with a true full-viewport layout (Pitfall 1) on notched/Dynamic-Island devices.
**Prevention:** Add `overscroll-behavior: contain` (or `none`) to scrollable containers and `overscroll-behavior-y: none` to `html,body`; add `viewport-fit=cover` to the viewport meta tag and pad the HUD/tab-bar/footer with `env(safe-area-inset-*)` once the layout goes full-viewport. Test specifically in installed-to-home-screen standalone mode on a real iOS device, not just Safari desktop responsive mode, since standalone-mode chrome differs from in-Safari behavior.
**Phase:** Shell/PWA phase, alongside the viewport/full-bleed layout work in Pitfall 1.

### Smart-quote-as-string-delimiter pattern is fragile to future content edits

Content strings are single-quoted JS string literals that deliberately use typographic curly quotes (`’ “ ”`) instead of straight ASCII quotes/apostrophes *inside* the string content (e.g. `h2='It’s good to see you again'`), which avoids needing to escape apostrophes. This is a legitimate technique, but it means any future content edit (by Josh, by an AI porting/regenerating a lesson, or by a well-meaning "fix the typography" pass that runs a find-and-replace normalizing curly quotes to straight quotes for "consistency") that introduces a straight `'` inside a single-quoted JS string will either throw a syntax error (page goes blank) or silently truncate the string mid-sentence depending on where it lands.
**Prevention:** If porting/editing lesson content programmatically, never run a blanket "normalize quotes" pass across the JS source without accounting for this; if anything, standardize on double-quoted JS string literals (or template literals) for new/ported content specifically to remove this fragility, while leaving Josh's already-verified files alone unless being actively edited.
**Phase:** Content-porting phase — establish the string-literal convention for the rebuild before the first lesson is ported, so it's consistent rather than mixed.

## Minor Pitfalls

### Honorific rendering convention drift

Content spells honorifics out in English ("peace be upon him," "Nuh, then Ibrahim...and finally Muhammad, peace be upon them all") rather than using the ﷺ Unicode glyph (U+FDFA ARABIC LIGATURE SALLALLAHOU ALAYHE WASSALLAM, or the simpler combined ﷺ character). This is consistent across the files reviewed. A rebuild that "polishes" this to the Unicode glyph for visual authenticity introduces a real font-coverage risk (that specific ligature glyph is unsupported by many non-Arabic-specialist fonts, including likely Inter/Poppins/Amiri at their loaded weights, and would need a fallback font stack of its own) and a consistency risk if only some instances get converted.
**Prevention:** Preserve the spelled-out convention as-is unless a deliberate, tested, full-pass decision is made to switch — don't let it drift inconsistently lesson-by-lesson.

### Node/lesson `id` stability underpins the entire progression + storage model

`stars{}` is keyed by lesson/review `id` (e.g. `'u1m1'`, `'u2m3b'`), and `nodeState()` computes locked/available/done purely by scanning the `UNITS[].nodes` array order and checking whether prior non-chest node ids have stars. If a rebuild renames ids, reorders `UNITS[].nodes`, or restructures the data shape (e.g., nesting a Part I/Part II split lesson as sub-steps of one node instead of two independent flat entries — the pattern Josh explicitly chose in v1.5 and documented as deliberate for future multi-part courses), any existing localStorage progress silently orphans (nodes that were "done" appear locked or vice versa) and the unlock-order logic must be re-verified from scratch.
**Prevention:** Treat lesson/review `id` values and the flat node-array ordering as a stable contract once set; if restructuring is necessary, write a small migration in `AW.S` rather than assuming a clean slate.

### Repeated, non-memoized `AW.state()`/`localStorage` reads during a single render

`learn.html`'s `render()` calls `AW.state()` (which does 5 separate `localStorage.getItem` + `JSON.parse` calls) multiple times per render pass, including once per node inside the `forEach`. At 15 lessons + 4 reviews this is a non-issue performance-wise today, but it's a pattern (re-reading global state repeatedly instead of reading once and passing it down) that would degrade if the course library grows (Fiqh/Seerah/Qur'an are named as future "coming soon" courses) or if state reads move to a slower backing store (e.g., if v2 adds account sync).
**Prevention:** When touching `render()`, read state once per render pass and thread it through, rather than propagating the repeated-read pattern into new code.

### First-time PWA service-worker cache versioning

The PWA/offline-caching requirement is new work (Josh's MVP has no manifest or service worker at all). The most common first-time PWA mistake is a service worker that caches pages/assets without a cache-busting version scheme, so updates to lesson content or engine fixes silently fail to reach already-installed users (stale cache serves old, possibly-buggy content indefinitely).
**Prevention:** Use a versioned cache name (bump on every deploy) with an explicit `activate` handler that purges old cache versions, and keep the caching strategy simple (network-first or stale-while-revalidate for HTML/data files, cache-first only for fonts/icons/static assets) given how frequently lesson content itself may still be revised pre-scholar-approval.
